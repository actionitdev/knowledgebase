# Performance Testing

Created by Yunkai (opens new window)

The following documentation explains the approach to conduct the performance test using Google Lighthouse and how to integrate it with the pipeline dashboard.

# Considerations

When doing a performance test on Solferino's website, we need to take the user's usage condition into consideration.

  1. Network condition: Potential users of Solferino's website are likely to suffer from the low bandwidth
  2. Device issues: Potential users of Solferino's website might use different ways to access the network

# Technical Solution - Google Lighthouse

2021S1 performance team suggested using Tauras (opens new window) as a solution to measure the performance. However, it concentrates on the server side performance while the client is more concerned with the client/browser side performance. As a result, Google Lighthouse is selected as the tool to use. Why Taurus/Jmeter is not suitable for client side performance (opens new window)

Lighthouse is an open-source, automated tool for improving the quality of web pages. You can run it against any web page, public or requiring authentication. It has audits for performance, accessibility, progressive web apps, SEO and more. In the project, it is only used to evaluate performance.

Following are some references used for writing the node scripts to run lighthouse.

  1. Getting started (opens new window) with Lighthouse overview.
  2. Lighthouse can be run programmatically as a node module (opens new window).
  3. Network throttling (opens new window) for simulating low bandwidth network conditions.
  4. Reporting (opens new window) for understanding the generated report.
  5. LHR (opens new window) contains all the audit information determined about the target web page. You can make use of it to do further analysis.

# Technical Solution - Integration with the dashboard

Pipeline team created a react web application dashboard (opens new window) to allow users to run everything in one place. Google Lighthouse is integrated with it to enable users to run performance tests on any given page with a button click.

Following are some references used for integration.

  1. Create component (opens new window) for the performance page.
  2. Make post call to node server (opens new window) and handling the api call in node server (opens new window).
  3. Send html report to the frontend by making the public folder available to the client and put the report in this folderRefer to this document (opens new window).

# Roadmap

It's assumed that further functionalities will still be integrated with the dashboard. Following are some suggestions for enhancing the performance test.

  1. Lighthouse configuration (opens new window) could be used to customize Lighthouse to suit your use case. For example, onlyCategories could be added to the config file to include only the specified categories in the final report to reduce the time taken by the test.
  2. If there are any previous tests on the same url, a comparison with the most recent report will be automatically conducted. The comparison report shows how metrics change in percentage. This is an existing function but comparing any two reports has not been implemented in the dashboard. The code for doing it is already in the script. You need to let users run it from the frontend and allow users to select two existing reports.
  3. The comparison report is in plain text now. Visualize the data to present to the user more friendly.Visualization JavaScript library (opens new window)
  4. Integrate Google Lighthouse into CI/CD pipeline (opens new window)
  5. Some settings need to be configured in the script. Provide UI in dashboard for non-technical users to change them easily.
